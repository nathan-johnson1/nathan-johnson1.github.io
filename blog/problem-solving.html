Problem Solving Reflection

1. A time I was blocked on a simple problem

I was trying to return "FlipFlop" for numbers divisible by 3 and 5, but my code was returning "Flip" instead. I used console logging, read the test expectations again, and talked through my logic out loud. I felt a bit stuck at first, but once I slowed down, I figured it out. I learned that the order of conditions really matters when writing if/else logic.

2. A time I elegantly solved a problem

I had to write a function to capitalize the first letter of each word in a sentence. I used pseudocode to plan it, then used .split(), .map(), and .join() to write a clean one-liner. It worked first try, and I felt confident and focused while writing it. I learned how useful planning is — it can really speed things up.

3. Reflecting on my confidence with problem-solving techniques

  • Pseudocode: Confident. Helps me plan and think clearly before coding.
  • Trying something: Comfortable. Experimenting often leads me toward a solution.
  • Rubber ducky method: Surprisingly effective! Talking through it helps me catch mistakes.
  • Reading error messages: Getting better. Slowing down and actually reading them helps a lot.
  • Console.logging: Very confident. It’s my default tool for figuring out what’s going on.
  • Googling: Confident. Great for syntax and checking method usage.
  • Asking peers for help: Still building confidence, but I know it’s valuable when I do.
  • Asking coaches for help: Sometimes nervous, but always helpful when I go for it.
  • Improving my process with reflection: Reflection helps me notice patterns and improve over time.

4. A time I was reluctant to ask for help

I once spent about 45 minutes stuck on a test that wouldn’t pass. I kept thinking I should be able to figure it out myself. Eventually, I asked for help and it turned out to be a small typo. Next time, I’ll give myself a 15–20 minute limit, if I’m still stuck, I’ll reach out. I’m learning that asking isn’t a weakness — it speeds up the learning process.
